Fine-Tuning Generation Models

A Comprehensive Guide Using TinyLlama with LoRA and Quantization

Introduction

This comprehensive guide demonstrates how to fine-tune TinyLlama-1.1B-Chat-v1.0 using LoRA (Low-Rank Adaptation) and Supervised Fine-Tuning (SFT). The process uses efficient parameter-efficient fine-tuning (PEFT) techniques to adapt a pre-trained language model to your specific task with minimal computational resources.

Hardware Tested: NVIDIA Jetson Orin Nano 8GB (CUDA 12.6, JetPack 6.4.7)
Training Time: ~22 minutes (1331 seconds) for 1 epoch on 3,000 examples
Model Size: 1.1 billion parameters (only 563,200 trainable with LoRA - 0.05%)
Memory Usage: ~4.5GB GPU memory during training
Training Speed: ~1.77 seconds/iteration, 2.25 samples/second
Final Training Loss: 1.52

Data Preparation (sft_dataprep.py)
Load and format conversational data from HuggingFace UltraChat 200k dataset. The data is transformed using apply_chat_template to match TinyLlama's expected format.
Dataset: HuggingFaceH4/ultrachat_200k (test_sft split, 3,000 samples)
# File: sft_dataprep.py from datasets import load_dataset from transformers import AutoTokenizer def load_tokenizer_model(model_name): """Load the tokenizer from the model""" tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True) return tokenizer def define_dataset(dataset_name, batch_size=1, shuffle=True): """Load UltraChat dataset with 3,000 samples""" dataset = ( load_dataset(dataset_name, split="test_sft") .shuffle(seed=42) .select(range(3_000)) # Only 3k examples for faster training ) return dataset def format_prompt(example, template_tokenizer): """Format conversational data using chat template""" chat = example["messages"] # apply_chat_template adds special tokens: <|user|>, <|assistant|> prompt = template_tokenizer.apply_chat_template(chat, tokenize=False) return {"text": prompt} def data_mapping(dataset, template_tokenizer): """Map dataset to text format for SFT""" dataset = dataset.map( lambda example: format_prompt(example, template_tokenizer), batched=False, remove_columns=dataset.column_names # Remove original columns ) print(dataset["text"][0]) # Show first example return dataset class DataPrep: """Class to prepare the data for the model""" def __init__(self): print("Starting Data Preparation") self.tokenizer = load_tokenizer_model("TinyLlama/TinyLlama-1.1B-Chat-v1.0") self.dataset = define_dataset("HuggingFaceH4/ultrachat_200k") self.data_mapping = data_mapping(self.dataset, self.tokenizer) self.data_mapping.save_to_disk("data_mapping")
Output Example: Formatted text with special tokens: 
<|user|>
Write a compelling mystery story set in a vineyard...</s>
<|assistant|>
Detective Jameson had been called to the vineyard...</s>
Key Points:
  • apply_chat_template: Automatically formats multi-turn conversations
  • tokenize=False: Returns text strings (not token IDs)
  • remove_columns: Removes original columns, keeps only "text"
  • 3,000 samples: Sufficient for testing, takes ~22 minutes to train
2
Load Model in FP16 (sft_model_tokenization.py)
Load TinyLlama model using PyTorch native FP16 precision (NOT 4-bit quantization). BitsAndBytes does NOT work on Jetson Orin Nano with CUDA 12.6 ARM64.
Critical for Jetson: Use torch.float16 instead of BitsAndBytes 4-bit. FP16 reduces model from 4.4GB to 2.2GB and is fully supported on Jetson.
# File: sft_model_tokenization.py import torch from transformers import AutoModelForCausalLM, AutoTokenizer def model_select(model_name): """Load model in FP16 precision""" model = AutoModelForCausalLM.from_pretrained( model_name, torch_dtype=torch.float16, # FP16 precision (NOT 4-bit!) device_map="auto", # Auto GPU placement low_cpu_mem_usage=True # Minimize CPU memory during load ) return model def tokenizer_select(model_name): """Load and configure tokenizer""" tokenizer = AutoTokenizer.from_pretrained( model_name, trust_remote_code=True ) # Set padding token to EOS token (TinyLlama has no PAD token) tokenizer.pad_token = tokenizer.eos_token tokenizer.padding_side = "right" # Pad on right for training return tokenizer class SFTModelTokenization: def __init__(self, model_name): print("Setting up Model for SFT") self.model = model_select(model_name) print(self.model.hf_device_map) # Show device placement print("Setting up Tokenizer for SFT") self.tokenizer = tokenizer_select(model_name)
Memory Usage:
  • FP32 (full precision): 4.4GB GPU memory
  • FP16 (half precision): 2.2GB GPU memory ✅
  • 4-bit (BitsAndBytes): NOT supported on Jetson ❌
Why FP16 instead of 4-bit?
  • BitsAndBytes compatibility: Doesn't work with CUDA 12.6 on ARM64 (Jetson)
  • Performance: FP16 is fully supported by PyTorch and runs well on Jetson
  • Memory: 2.2GB for model + 1.5GB for training = 3.7GB (fits in 8GB)
  • Accuracy: FP16 maintains better precision than 4-bit
⚠️ Important: Close Cursor IDE before loading the model! Cursor uses 2.5-3.5GB GPU memory, leaving insufficient memory for model loading. Develop code in Cursor, save, close Cursor, then run training.
3
Apply LoRA Configuration (sft_lora_config.py)
LoRA (Low-Rank Adaptation) adds small trainable adapter layers to the model. This reduces trainable parameters from 1.1 billion to only 563,200 (0.05%)!
Efficiency: LoRA allows fine-tuning with minimal memory and compute, making it perfect for edge devices like Jetson Orin Nano.
# File: sft_lora_config.py from peft import LoraConfig, get_peft_model def apply_lora(model, l_alpha, l_dropout, l_rank, l_bias, l_task_type, l_target_modules): print("Setting up lora config") peft_config = LoraConfig( lora_alpha=l_alpha, # 8 (scaling factor) lora_dropout=l_dropout, # 0.0 (no dropout) r=l_rank, # 4 (rank - low for memory efficiency) bias=l_bias, # "none" (don't train bias) task_type=l_task_type, # "CAUSAL_LM" target_modules=l_target_modules # ["q_proj", "v_proj"] only ) # NO prepare_model_for_kbit_training() - we're not using 4-bit! model = get_peft_model(model, peft_config) return model, peft_config
Actual Configuration Used:
# From config.py ALPHA = 8 # LoRA scaling (alpha/r = 2.0) DROPOUT = 0.0 # No dropout for stability R = 4 # Low rank for memory BIAS = "none" # Don't train bias TASK_TYPE = "CAUSAL_LM" # Causal language modeling TARGET_MODULES = ["v_proj", "q_proj"] # Only attention query/value
Training Output: 
trainable params: 563,200 || all params: 1,100,611,584 || trainable%: 0.0512

LoRA adapters are added to all 22 transformer layers (q_proj and v_proj in each layer)

Why Only q_proj and v_proj?
  • Memory constraints: Jetson has only 8GB, need to minimize trainable params
  • Attention is key: Query/Value projections capture most of the learning
  • 563k params: Still effective for fine-tuning on specific tasks
  • Could expand: Add k_proj, o_proj, gate_proj if more GPU memory available
LoRA Rank (r) Explained:
  • r=4: 563k trainable params, lowest memory, fast training
  • r=8: ~1M trainable params, better capacity
  • r=16: ~2M trainable params, highest quality but more memory
  • Your choice: r=4 is optimal for 8GB Jetson

Training Arguments (sft_trainer_config.py)
Configure training hyperparameters optimized for Jetson Orin Nano 8GB. Uses FP16 mixed precision, gradient checkpointing, and aggressive memory optimization.
# File: sft_trainer_config.py from transformers import TrainingArguments from trl import SFTTrainer def define_training_args(output_dir, per_dev_train_batch_size, grad_acc_step, optim_method, lr, lr_sched_type, num_of_epochs, log_steps, fp16_bool_value, grad_checkpoint_bool_value): training_arguments = TrainingArguments( output_dir=output_dir, # "./results" per_device_train_batch_size=per_dev_train_batch_size, # 1 gradient_accumulation_steps=grad_acc_step, # 4 (effective batch = 4) optim=optim_method, # "adamw_torch" learning_rate=lr, # 2e-4 lr_scheduler_type=lr_sched_type, # "cosine" num_train_epochs=num_of_epochs, # 1 logging_steps=log_steps, # 10 fp16=fp16_bool_value, # True (FP16 mixed precision) gradient_checkpointing=grad_checkpoint_bool_value # True (save memory) ) return training_arguments def sft_trainer(model, dataset, tokenizer, training_args, max_seq_len, peft_config): trainer = SFTTrainer( model=model, train_dataset=dataset, dataset_text_field="text", # Column from data prep tokenizer=tokenizer, args=training_args, max_seq_length=max_seq_len # 64 tokens (memory efficient) ) return trainer
Actual Configuration (from config.py):
OUTPUT_DIR = "./results" PER_DEV_TRAIN_BATCH_SIZE = 1 # Physical batch size GRAD_ACC_STEP = 4 # Effective batch = 1 × 4 = 4 OPTIM_METHOD = "adamw_torch" # Standard AdamW LR = 2e-4 # Learning rate for LoRA LR_SCHED_TYPE = "cosine" # Cosine annealing NUM_OF_EPOCHS = 1 # Single epoch for testing LOG_STEPS = 10 # Log every 10 steps FP16_BOOL_VALUE = True # FP16 mixed precision ✅ GRAD_CHECKPOINT_BOOL_VALUE = True # Gradient checkpointing ✅ MAX_SEQ_LEN = 64 # Short sequences for memory
Key Settings Explained:
  • Batch size = 1: Minimum to fit in 8GB GPU memory
  • Gradient accumulation = 4: Simulates batch size of 4 without extra memory
  • FP16: Mixed precision training (2x faster, 2x less memory)
  • Gradient checkpointing: Saves ~40% GPU memory at cost of 20% speed
  • Cosine scheduler: Gradually reduces learning rate (2e-4 → 0)
  • max_seq_length=64: Short sequences use less memory (memory scales as O(n²))
Training Results:
  • Total steps: 750 (3000 samples ÷ 4 effective batch)
  • Speed: 1.77 seconds per iteration
  • Total time: 1331 seconds (~22.2 minutes)
  • Throughput: 2.25 samples/second
  • GPU memory: ~4.5GB during training

Train the Model (lora_finetune.py)
Execute the complete training pipeline: data prep → model loading → LoRA setup → training.
# File: lora_finetune.py (main orchestration script) import os os.environ["PYTORCH_CUDA_ALLOC_CONF"] = "expandable_segments:True" os.environ["TOKENIZERS_PARALLELISM"] = "false" import torch import config from sft_dataprep import DataPrep from sft_model_tokenization import SFTModelTokenization from sft_lora_config import apply_lora from sft_trainer_config import define_training_args, sft_trainer if __name__ == "__main__": # Step 1: Prepare data data_prep = DataPrep() dataset_transformed = data_prep.data_mapping # Step 2: Load model and tokenizer in FP16 sft_model_tokenization = SFTModelTokenization( model_name=config.MODEL_NAME ) model_sft = sft_model_tokenization.model tokenizer_sft = sft_model_tokenization.tokenizer # Step 3: Apply LoRA model, peft_config = apply_lora( model_sft, config.ALPHA, config.DROPOUT, config.R, config.BIAS, config.TASK_TYPE, config.TARGET_MODULES ) model.enable_input_require_grads() model.print_trainable_parameters() # Output: trainable params: 563,200 || all params: 1,100,611,584 # Step 4: Setup training arguments training_arguments = define_training_args( config.OUTPUT_DIR, config.PER_DEV_TRAIN_BATCH_SIZE, config.GRAD_ACC_STEP, config.OPTIM_METHOD, config.LR, config.LR_SCHED_TYPE, config.NUM_OF_EPOCHS, config.LOG_STEPS, config.FP16_BOOL_VALUE, config.GRAD_CHECKPOINT_BOOL_VALUE ) # Step 5: Create SFTTrainer and train print("Starting Training", flush=True) sft_trainer_obj = sft_trainer( model, dataset_transformed, tokenizer_sft, training_arguments, config.MAX_SEQ_LEN, peft_config ) train_result = sft_trainer_obj.train() print(train_result, flush=True) # Training takes ~22 minutes, loss goes from 1.82 → 1.52

Why Save Adapters Separately?
  • Efficiency: Adapters are only 2.2MB vs 2.2GB full model
  • Modularity: Can load different adapters for different tasks
  • Sharing: Easy to share fine-tuned weights without uploading full model
  • Multi-task: Train multiple adapters for one base model

Merge Adapters with Base Model (sft_merge_model_weights.py)
Merge the trained LoRA adapters back into the base model to create a standalone fine-tuned model.
# File: sft_merge_model_weights.py from peft import AutoPeftModelForCausalLM def get_merged_model(): # Load model with adapters model = AutoPeftModelForCausalLM.from_pretrained( "TinyLlama-1.1B-qlora", low_cpu_mem_usage=True, device_map="cpu" # Load on CPU for merging ) # Merge and unload adapters merged_model = model.merge_and_unload() return merged_model # Usage in main script: print("Preparing the LoRA Model with merged weights", flush=True) merged_model_final = get_merged_model() print("Saving the merged model", flush=True) merged_model_final.save_pretrained("TinyLlama-1.1B-merged") # Files saved (~2.2GB): # TinyLlama-1.1B-merged/ # ├── config.json # ├── generation_config.json # └── model.safetensors (2.2GB - base + adapters merged)
Merge Complete! You now have:
  • TinyLlama-1.1B-qlora/: LoRA adapters only (2.2MB) - for loading with base model
  • TinyLlama-1.1B-merged/: Full merged model (2.2GB) - standalone, ready for inference
When to Use Each:
  • Use adapters: If you want to swap multiple LoRA adapters with one base model
  • Use merged model: For deployment, inference, or sharing a complete model
  • Merged model: Slightly faster inference (no adapter overhead)

Merge Adapter with Base Model
After training, merge the LoRA adapters with the base model to create a standalone model that doesn't require PEFT dependencies for inference.
Why Merge? Merged models are easier to deploy, have faster inference, and don't require the PEFT library at runtime.
# Load the fine-tuned model with adapters from peft import PeftModel from transformers import AutoModelForCausalLM, AutoTokenizer # Load base model (in full precision for merging) base_model_name = "TinyLlama/TinyLlama-1.1B-Chat-v1.0" base_model = AutoModelForCausalLM.from_pretrained( base_model_name, torch_dtype=torch.float16, device_map="auto", ) # Load PEFT model (with LoRA adapters) peft_model = PeftModel.from_pretrained( base_model, "./tinyllama-finetuned-final" ) # Merge adapters into base model print("Merging LoRA adapters with base model...") merged_model = peft_model.merge_and_unload() # Save the merged model merged_model.save_pretrained("./tinyllama-merged") tokenizer.save_pretrained("./tinyllama-merged") print("Merged model saved to ./tinyllama-merged") # Optional: Push to Hugging Face Hub # merged_model.push_to_hub("your-username/tinyllama-finetuned") # tokenizer.push_to_hub("your-username/tinyllama-finetuned")
Merge Process:
  • LoRA adapter weights are added to the corresponding base model layers
  • The result is a single model with updated weights
  • Merged model can be loaded like any standard Hugging Face model
  • No PEFT library needed for inference after merging

Use SFT Model with Prompts
Test your fine-tuned model by generating responses to prompts. This validates that the fine-tuning was successful and the model learned the desired behavior.
# Load the merged model for inference from transformers import AutoModelForCausalLM, AutoTokenizer, pipeline # Load model and tokenizer model = AutoModelForCausalLM.from_pretrained( "./tinyllama-merged", torch_dtype=torch.float16, device_map="auto", ) tokenizer = AutoTokenizer.from_pretrained("./tinyllama-merged") # Create a text generation pipeline pipe = pipeline( "text-generation", model=model, tokenizer=tokenizer, max_new_tokens=256, temperature=0.7, top_p=0.95, do_sample=True, ) # Test with a prompt prompt = """### Instruction: Explain the concept of neural networks in simple terms. ### Input: ### Response: """ # Generate response result = pipe(prompt) print(result[0]['generated_text']) # Example 2: Multiple prompts test_prompts = [ "Write a Python function to reverse a string.", "What are the benefits of using LoRA for fine-tuning?", "Explain gradient descent optimization." ] for prompt in test_prompts: formatted_prompt = f"""### Instruction: {prompt} ### Input: ### Response: """ output = pipe(formatted_prompt) print(f"\n{'='*60}") print(f"Prompt: {prompt}") print(f"{'='*60}") print(output[0]['generated_text'])
Inference Tips:
  • Temperature: Lower (0.3-0.7) for focused responses, higher (0.8-1.2) for creativity
  • Top-p (nucleus sampling): 0.9-0.95 balances quality and diversity
  • max_new_tokens: Control response length (128-512 typical)
  • Prompt format: Use the same format as training for best results
# Advanced: Batch inference for efficiency def batch_generate(prompts, max_length=256): inputs = tokenizer(prompts, return_tensors="pt", padding=True).to(model.device) outputs = model.generate( **inputs, max_new_tokens=max_length, temperature=0.7, top_p=0.95, do_sample=True, pad_token_id=tokenizer.eos_token_id, ) return [tokenizer.decode(output, skip_special_tokens=True) for output in outputs] # Test batch generation batch_prompts = [format_instruction(p) for p in test_prompts] results = batch_generate(batch_prompts) for prompt, result in zip(test_prompts, results): print(f"\nPrompt: {prompt}") print(f"Response: {result}")

Hardware Requirements

  • Minimum: NVIDIA GPU with 8GB VRAM (e.g., RTX 3060)
  • Recommended: 16GB+ VRAM (e.g., RTX 4090, A100)
  • RAM: 16GB+ system RAM
  • Storage: 20GB+ free space for models and checkpoints

Common Issues & Solutions

Out of Memory (OOM):
  • Reduce batch size to 1 or 2
  • Enable gradient checkpointing
  • Lower max_seq_length to 256 or 384
  • Use smaller LoRA rank (r=8)
Poor Model Performance:
  • Increase training data quality and quantity
  • Train for more epochs (3-5)
  • Adjust learning rate (try 1e-4 to 5e-4)
  • Increase LoRA rank for more capacity

Jetson Orin Nano Specific Issues

Issue: Cursor IDE Using GPU Memory (2.5-3.5 GB)
  • Problem: Cannot load TinyLlama model while Cursor is running
  • Error: RuntimeError: NVML_SUCCESS INTERNAL ASSERT FAILED
  • Solution: Close Cursor before running training scripts
  • Workflow: Develop in Cursor → Save → Close Cursor → Run training from terminal
  • Memory breakdown: Cursor (3GB) + Model (2.2GB) + Training (1.5GB) = 6.7GB > 8GB available
Issue: GPU Locked at Low Frequency (612 MHz instead of 1020 MHz)
  • Symptoms: GPU showing 612 MHz max frequency, training 2-3x slower
  • Root cause: jetson_clocks may not work properly on JetPack 6.x
  • Solution: Manually set GPU frequency
# Check available GPU frequencies cat /sys/devices/platform/bus@0/17000000.gpu/devfreq/17000000.gpu/available_frequencies # Output: 306000000 408000000 510000000 612000000 714000000 816000000 918000000 1020000000 # Set GPU to maximum frequency (1020 MHz) echo 1020000000 | sudo tee /sys/devices/platform/bus@0/17000000.gpu/devfreq/17000000.gpu/max_freq echo 1020000000 | sudo tee /sys/devices/platform/bus@0/17000000.gpu/devfreq/17000000.gpu/min_freq # Or change governor to performance echo performance | sudo tee /sys/devices/platform/bus@0/17000000.gpu/devfreq/17000000.gpu/governor # Verify cat /sys/devices/platform/bus@0/17000000.gpu/devfreq/17000000.gpu/cur_freq # Should show: 1020000000

2. Data Preparation Module (sft_dataprep.py)

Key Insight: Uses apply_chat_template to format conversational data into TinyLlama's expected format. The dataset uses only 3,000 examples from UltraChat 200k for faster training.
from datasets import load_dataset from transformers import AutoTokenizer def format_prompt(example, template_tokenizer): """Format conversational data using chat template""" chat = example["messages"] prompt = template_tokenizer.apply_chat_template(chat, tokenize=False) return {"text": prompt} def data_mapping(dataset, template_tokenizer): """Transform dataset to text format for SFT""" dataset = dataset.map( lambda example: format_prompt(example, template_tokenizer), batched=False, remove_columns=dataset.column_names # Remove original columns ) return dataset class DataPrep: def __init__(self): print("Starting Data Preparation") self.tokenizer = AutoTokenizer.from_pretrained( "TinyLlama/TinyLlama-1.1B-Chat-v1.0" ) self.dataset = ( load_dataset("HuggingFaceH4/ultrachat_200k", split="test_sft") .shuffle(seed=42) .select(range(3_000)) # Only 3k examples ) self.data_mapping = data_mapping(self.dataset, self.tokenizer) self.data_mapping.save_to_disk("data_mapping")

4. LoRA Configuration (sft_lora_config.py)

Efficiency: LoRA reduces trainable parameters from 1.1B to only 563,200 (0.05%)! This makes training feasible on consumer hardware.

9. Key Learnings & Best Practices

What Worked:
  • FP16 instead of 4-bit: BitsAndBytes doesn't work on Jetson; PyTorch native FP16 is perfect
  • Small LoRA rank (r=4): Sufficient for fine-tuning, keeps memory low
  • Gradient accumulation: Effective batch size of 4 with physical batch of 1
  • Short sequences (64 tokens): Dramatically reduces memory footprint
  • Gradient checkpointing: Essential for 8GB GPU memory
  • Closing Cursor IDE: Frees 2.5-3.5GB GPU memory for training
  • GPU at 1020 MHz: 2-3x faster training than 612 MHz
  • Modular code structure: Easy to debug and modify
Common Pitfalls to Avoid:
  • Don't use BitsAndBytes on Jetson: CUDA 12.6 ARM64 incompatibility
  • Don't forget to close Cursor: It uses 3GB+ GPU memory
  • Don't skip GPU frequency tuning: Massive performance impact
  • Don't use NumPy 2.x: Causes dependency conflicts
  • Don't use Transformers 4.47.x: Incompatible with TRL 0.12.2
  • Don't set max_seq_length too high: Memory usage scales quadratically

Key Optimizations Explained

1. TTY vs GUI:
  • GNOME Desktop Manager (gdm): Uses ~1.5GB GPU memory
  • Xorg + gnome-shell: Uses ~500MB GPU memory
  • Running from TTY frees ~2GB total GPU memory
2. Memory Tuning:
  • vm.overcommit_memory=1: Allows kernel to allocate more memory than physically available
  • drop_caches: Clears file system cache to free memory
  • PYTORCH_CUDA_ALLOC_CONF: Optimizes PyTorch memory allocator
3. Background Process Cleanup:
  • Stops unnecessary services (Docker, Evolution, GNOME Software)
  • Kills desktop portals and indicators
  • Frees CPU cycles and memory for training
